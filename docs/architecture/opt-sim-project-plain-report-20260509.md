# Opt-Sim 项目通俗化解读报告

## 说明

这份报告的目标不是写成论文，也不是写成只给算法工程师看的技术笔记，而是尽量用最通俗、最直白、最容易建立整体感觉的语言，把 `opt-sim` 这个项目讲清楚。

项目里现在有两条主线：

1. 一条是“算法和物理仿真”主线，也就是系统到底在算什么、为什么这样算、算出来的东西代表什么。
2. 一条是“软件工程化”主线，也就是这些算法能力是怎样被包成一个能运行、能调用、能追踪、能展示、能持续迭代的软件系统。

如果只看第一条线，你会知道它“会算”；如果只看第二条线，你会知道它“像个系统”。只有两条线合在一起，才是这个项目真正的样子：它不是一个单独的公式脚本，也不是一个只会摆页面的前后端壳子，而是一个把“物理模型、AI 逆向设计、实验复现、工作流、界面、文档、测试、产物管理”串起来的原型平台。

下面我会分成两大部分来讲：

- 第一部分：算法和物理机理仿真等部分都在做什么。
- 第二部分：软件工程化部分都做什么。

在讲细节之前，先用一句最通俗的话概括这个项目：

**这个项目想做的是：给它一个想要的光学效果，比如某种结构色，或者某种全息显示目标，它帮你反推出可能的物理结构参数，再用物理仿真验证这些参数靠不靠谱，最后把结果组织成一个可审阅、可追踪、可导出的设计工作台。**

换句话说，这不是“从材料算颜色”的单向软件，而是“从目标效果反推结构”的逆向设计软件。

---

## 一、先用大白话讲：这个项目到底想解决什么问题

### 1.1 它面对的不是普通图片处理问题，而是“光学设计反推”问题

很多软件的问题是这样的：

- 输入一张图，输出另一张图；
- 输入一些表格，输出一个分类结果；
- 输入一段文字，输出一段总结。

但这个项目不是。

这个项目面对的问题更像这样：

- 我想要一种视觉效果，比如暖铜色、蓝绿色、某种角度会变色的结构色；
- 或者我想要一种全息重建结果，比如目标图案在某个光学系统中被显示出来；
- 现在请你告诉我，真实世界里应该做出什么结构、什么膜层厚度、什么相位图，才有可能实现这种效果。

这就叫“逆向设计”。

所谓逆向设计，意思不是先知道结构再算结果，而是先知道想要的结果，再反过来猜测和筛选可能的结构。

这类问题比普通预测难，主要有三个原因：

1. 它不是一对一关系。
   一个目标颜色，往往可能对应很多套物理结构。也就是说，答案不是唯一的。
2. 它不能只看“像不像”。
   就算一种结构在理想计算里颜色很准，也可能制造起来太脆弱，一点点厚度误差就跑色了。
3. 它必须受物理规律约束。
   不是神经网络随便编一个答案就行，最后还得回到光学仿真里验一遍。

所以这个项目的基本思想不是“让 AI 直接拍脑袋出结论”，而是：

- 先用物理模型定义什么是合理的；
- 再用 AI 提高逆向搜索效率；
- 最后用工程系统把候选方案组织起来，给人做审阅和确认。

### 1.2 它现在实际上有两种模式

从当前代码和文档看，这个项目已经不是单一模式了，而是分成两条业务路线：

1. `structural-color`
   结构色薄膜路线，也就是 Ag-SiO2-Ag 薄膜结构的颜色逆向设计。
2. `neural-holography`
   神经全息路线，也就是围绕 CGH、相位型 SLM、CITL、HoloNet 这些概念做的全息显示规划路线。

这两条路线共享同一个软件平台，但底层算法并不相同。

你可以把它们理解为：

- 结构色路线是在问：“做什么膜层厚度，能得到目标颜色？”
- 全息路线是在问：“做什么相位图，能重建目标图像？”

两者都属于“光学逆向设计”，但一个偏薄膜干涉，一个偏波前传播。

### 1.3 为什么项目里会同时有物理仿真和 AI

很多人第一次看这类项目会有个疑问：

“既然已经有物理公式了，为什么还要 AI？”

答案很简单：

- 物理模型负责“算得对”；
- AI 负责“找得快”；
- 工程系统负责“用得起来”。

如果没有物理模型，AI 的答案不可信；
如果没有 AI，逆向搜索会很慢，尤其在设计空间大时；
如果没有工程系统，算法即使有效，也难以组织成稳定工作流。

所以这三者在这个项目里不是互相替代，而是分工配合。

## 2.1 结构色路线的核心思想：从颜色反推膜层厚度

### 2.1.1 结构色不是“涂料颜色”，而是“结构导致的颜色”

普通颜色常常来自染料、颜料，也就是材料本身吸收某些波长、反射某些波长。

结构色不是这样。

结构色更多是因为材料的微观结构、膜层厚度、折射率差异等因素，让不同波长的光发生：

- 干涉；
- 增强；
- 抵消；
- 角度相关变化。

最后，人眼看到的颜色，不完全由“是什么材料”决定，而是很大程度由“材料怎么排布、厚度是多少、光怎么走”决定。

这个项目当前的结构色路线，采用的是比较典型的一种简化模型：

- 底部银层 `Ag`
- 中间二氧化硅 `SiO2`
- 顶部银层 `Ag`

也就是一个 `Ag-SiO2-Ag` 的三层结构。

它不是把真实制造过程全部塞进来，而是先抓住最关键的颜色形成机制：**多层薄膜干涉**。

### 2.1.2 这个结构为什么适合做项目起点

因为它有几个优点：

1. 参数少。
   只要关心三层厚度：底银厚度、中间介质厚度、顶银厚度。
2. 物理模型成熟。
   这类分层介质用传输矩阵法就能算，理论基础非常稳定。
3. 颜色结果明显。
   厚度变化会带来可观的透射或反射光谱变化，进而变化成不同颜色。
4. 能做论文复现。
   当前项目明确对齐了一篇 cGAN 结构色逆向设计论文，因此这个三层模型既有工程意义，也有复现实验意义。

换句话说，它既是一个“足够真实”的问题，又是一个“还算可控”的问题。

### 2.1.3 这条路线里最根本的输入和输出是什么

如果只看运行时逆向设计接口，最基本的输入和输出可以这样理解：

输入：

- 一个目标颜色，比如 `#bf6f4f`
- 一个入射角，比如 `45.0` 度
- 一种偏振条件，比如 `TE`、`TM` 或非偏振光
- 希望返回多少个候选答案，比如 `topK = 3`

输出：

- 若干个候选结构
- 每个候选结构的三层厚度
- 每个候选结构算出来的模拟颜色
- 工艺扰动前后颜色变化
- 排名分数
- 制造稳健性描述
- 导出估算信息

所以它不是简单返回“最优厚度”，而是返回一个带解释的候选集合。

这点非常关键，因为项目从一开始就接受了一个现实：**逆向设计通常不是单解问题。**

## 2.2 薄膜正向仿真到底在算什么

### 2.2.1 所谓“正向仿真”，就是给定结构，算会出现什么颜色

逆向设计听起来很高级，但它必须建立在正向仿真之上。

正向仿真就是：

- 如果我已经知道膜层结构；
- 比如底部银 20nm，中间 SiO2 120nm，顶部银 18nm；
- 在某个入射角和偏振条件下；
- 最后它的透射光谱或反射光谱是什么；
- 再进一步，人眼会把它看成什么颜色。

也就是说，正向仿真负责回答：

**“这一套物理结构，会呈现出怎样的光学结果？”**

只有正向问题算得稳，逆向问题才有基础。

### 2.2.2 项目采用的核心物理方法：传输矩阵法 TMM

当前项目在 `backend/app/algorithms/optics.py` 里实现的核心，是传输矩阵法，也叫 Transfer Matrix Method，常写成 TMM。

通俗理解，它在做的事是：

1. 把每一层材料当成一个会影响光传播的“关卡”。
2. 每一层都有自己的折射率、厚度、相位延迟效应。
3. 光穿过这一层时，电场和磁场关系会发生变化。
4. 每一层都可以写成一个 2x2 的“特征矩阵”。
5. 多层堆叠时，把这些矩阵一个接一个乘起来。
6. 最后从总矩阵里推导出反射系数和透射系数。

简单说就是：

**单层的影响先写成小矩阵，多层的影响靠矩阵连乘叠加起来。**

这套方法的好处是：

- 理论成熟；
- 适合处理平面分层介质；
- 数值计算效率高；
- 对薄膜问题特别常见。

### 2.2.3 代码里具体考虑了哪些物理量

从实现上看，代码里至少明确考虑了这些因素：

1. 材料折射率
   - 空气
   - `SiO2`
   - `Ag`
   - 石英基底
2. 膜层厚度
   - 底银层厚度
   - 二氧化硅厚度
   - 顶银层厚度
3. 波长
   - 当前默认在 `380nm` 到 `780nm` 范围上取样
   - 步长是 `5nm`
4. 入射角
   - 支持法向入射
   - 也支持斜入射
5. 偏振
   - `TE`
   - `TM`
   - 非偏振光（即 TE 和 TM 平均）

这说明它不是一个“只会在 0 度算一次”的极简演示，而是已经把角度和偏振纳入到了运行时评价条件里。

### 2.2.4 为什么要区分 TE、TM 和非偏振

因为光在斜入射时，不同偏振态的传播行为并不完全一样。

如果用最通俗的话讲：

- `TE` 光和 `TM` 光在界面上的“过法”不同；
- 所以相同结构在相同角度下，TE 和 TM 的透射/反射结果会不同；
- 真实世界里的白光或普通照明，往往又不是严格单一偏振态；
- 所以如果想更贴近实际使用，有时候要用 TE、TM 平均得到非偏振结果。

项目里把这些条件暴露出来，本质上是在告诉用户：

“候选结果不是脱离使用场景的，它依赖观察条件。”

这比只报一个颜色值要真实得多。

### 2.2.5 从光谱到颜色：为什么不能直接输出一个 RGB

很多非光学背景的人会问：

“既然最后只是显示颜色，为什么不直接算 RGB？”

因为光学结构先产生的不是 RGB，而是**光谱分布**。

也就是说，正向仿真第一步算出来的通常是：

- 在不同波长下，透过了多少光；
- 或反射了多少光。

这是一条曲线，不是一个颜色块。

而人眼看到的颜色，是这条光谱曲线在特定照明条件、标准观察者模型下综合出来的视觉结果。

所以项目里做的是两步：

1. 先算透射光谱或反射光谱；
2. 再把光谱积分成颜色空间里的 `XYZ`、`Lab`、`sRGB`。

这也是为什么颜色部分被单独设计成一个明确模块，而不是随手写一个“RGB 近似公式”。

## 2.3 颜色学部分都做了什么

### 2.3.1 它不是简单“调色”，而是在做标准色度学换算

项目里一个很容易被低估的部分，是 `colorimetry_data.py` 和 `optics.py` 里的颜色学处理。

很多原型项目在这一层会偷懒，直接：

- 设一套假的颜色匹配曲线；
- 或者把几个波长粗暴映射成 RGB；
- 或者根本不区分照明条件。

但这个项目显然不是这样。

它用了真实的：

- D65 标准照明数据；
- 三刺激值函数数据；
- `colour-science` 库做标准 `XYZ -> Lab`、`XYZ -> sRGB` 变换；
- `Delta E 2000` 做颜色差评价。

这意味着项目在一个很关键的问题上是认真对待的：

**算法评价不能只看“看着像不像”，而要用标准颜色学方法说清楚“差多少”。**

### 2.3.2 D65 和三刺激值为什么重要

因为颜色不是物体自己“固有地长出来”的一个数字。

同样一条光谱，在不同照明下，看起来可能不一样。

所以项目里要指定：

- 光源是什么样；
- 标准观察者怎么看；
- 如何把整条光谱加权积分成人眼感知的颜色。

这里的 D65 可以粗略理解成一种标准日光照明参考。

三刺激值函数则可以粗略理解成人眼对不同波长响应的标准化模型。

只有把这两者结合起来，才有资格把仿真的光谱结果转换成标准颜色空间。

### 2.3.3 为什么项目强调 Lab，而不是只强调 RGB

因为 RGB 更适合显示，不适合度量差异。

如果只是为了网页上展示颜色块，RGB 很方便。

但如果要比较两个颜色是否接近，RGB 的距离不一定对应人眼真实感知差异。

所以项目里更看重 `Lab`，因为它是更接近人眼感知均匀性的颜色空间。

再进一步，项目不仅用 Lab，还用了 `Delta E 2000`。

最通俗理解：

- Lab 是颜色的坐标系；
- Delta E 是“两个颜色之间差多远”的量尺；
- `Delta E 2000` 是这个量尺里更贴近人眼感受的一种版本。

这会直接影响算法评价和候选排序的可信度。

### 2.3.4 为什么项目里专门做了颜色数据校验

`colorimetry_data.py` 不只是读文件，它还做了很多校验：

- 波长必须单调递增；
- 步长必须连续；
- D65 数据与三刺激表中的 D65 列要一致；
- 请求的波长范围不能超出支持区间；
- 数据会做缓存，避免重复加载。

从物理意义上讲，这样做是在减少“看不见的错误”。

因为颜色学数据如果悄悄错了，系统不一定会报错，但最终训练指标、候选排序、页面显示都会一起偏。

所以你会发现，这个项目在颜色学上不是把它当成附属小工具，而是把它当成算法可信度的一部分。

## 2.4 运行时逆向设计流程：用户发起一次结构色请求后，系统内部都做了什么

下面我们不讲理论，直接讲一次请求在系统里是怎么流动的。

假设用户提交一个请求：

- 目标颜色：`#bf6f4f`
- 角度：`45°`
- 偏振：`TM`
- 返回 3 个候选方案

那么系统做的事情，大致可以分成八步。

### 第一步：把目标颜色从十六进制转成 Lab

用户输入的是十六进制颜色，比如 `#bf6f4f`。

这对人和前端展示很方便，但对光学逆向设计来说，不能直接拿来比较。

所以系统首先会把这个颜色转成 Lab。

这一步相当于把“网页颜色写法”变成“颜色学比较坐标”。

### 第二步：建立检索网格

接下来，系统会在一个预设的厚度搜索空间里枚举样本。

当前网格范围大致是：

- 底银：`10nm` 到 `30nm`
- 中间 `SiO2`：`60nm` 到 `180nm`
- 顶银：`10nm` 到 `30nm`

并且按照一定步长采样。

每一个采样点其实就是一套候选膜层结构。

系统会对这个网格中的每一个结构做正向仿真，得到对应 Lab。

这样就形成了一个很重要的“样本库”：

- 左边是结构参数；
- 右边是对应颜色。

可以理解成，它先搭了一个“厚度到颜色”的离散地图。

### 第三步：先做检索，找最近的几个种子解

有了样本库之后，系统会先用检索法找离目标颜色最近的点。

当前支持两种检索度量：

1. `euclidean_lab`
   直接在 Lab 空间里算欧氏距离。
2. `delta_e_2000`
   用更接近人眼感知的 Delta E 2000 排序。

这一步很像：

“先别让 AI 天马行空，先从已知样本里挑几个看起来最像的。”

这几个检索到的结果，后面会作为基线候选或者局部优化起点。

### 第四步：再让 cGAN 生成更多候选结构

检索法有一个优点：稳。
但也有一个缺点：它只能在已有离散样本里找。

为了得到更多可能的解，系统还会调用 cGAN。

这里的 cGAN 作用不是替代物理，而是做一件很具体的事：

**根据目标颜色，生成一些可能的膜层厚度组合。**

换句话说，它在学一种分布：

- 输入：目标 Lab
- 输出：多组可能的三层厚度

而不是只输出唯一答案。

这点正好符合结构色逆向设计的“一对多”特征。

### 第五步：对检索结果和 cGAN 结果都做物理复验

这是项目特别重要的一步。

系统不会因为一个候选是 cGAN 生成出来的，就直接相信它。

所有候选，不管是：

- 检索来的；
- 还是 cGAN 生成的；

最后都要再进一次正向物理仿真。

也就是说，每个候选都要重新算：

- 模拟颜色；
- 与目标颜色的 Delta E；
- 在指定角度、指定偏振下的表现。

这一步相当于“AI 负责提议，物理模型负责盖章”。

### 第六步：做局部微调 refinement

系统还做了一步很实用的事：局部微调。

什么意思？

假设一个候选大致不错，但不是最优，系统不会马上放弃，而是围绕这个点做小范围搜索。

比如在当前厚度上下浮动一点点，看看是否能把颜色误差再压低一些。

这一步很像：

- 检索给你一个粗定位；
- cGAN 给你一个有创造性的猜测；
- 局部微调帮你做最后的小修小补。

所以最终候选，不是“原封不动的检索结果”或“原封不动的神经网络输出”，而是经过局部物理优化之后的结果。

### 第七步：做工艺扰动测试

项目很清楚一件事：

**实验室和工厂里，厚度不可能永远一丝不差。**

所以系统会对候选方案做一个简化的工艺敏感性检查：

- 给每层厚度加上 `+0.5nm`
- 再给每层厚度减去 `-0.5nm`
- 重新仿真颜色变化

这样它就能知道：

- 这个候选在理想状态下颜色误差是多少；
- 如果制造时略有偏差，颜色会不会突然飘得很厉害。

这就是代码里 `drift`、`Process drift` 一类指标的来源。

通俗地说，它在回答：

“这方案不只是看起来准不准，还要看它娇不娇气。”

### 第八步：综合评分、排序、打标签

最后系统会把这些信息综合起来，做排序。

它不是只按一个 `Delta E` 排，而是用一个综合分数。

这个分数可以粗略理解成：

- 基础颜色误差越小越好；
- 工艺漂移越小越好；
- 排名越靠前越值得优先推荐。

之后系统会给候选打上比较容易理解的状态：

- `Recommended`
- `Robust`
- `Watch`

这些词的真实含义是：

- `Recommended`：综合最好，最值得先试；
- `Robust`：不一定最优，但更稳定；
- `Watch`：看上去可用，但对工艺误差比较敏感。

这一步其实就是把一堆技术指标翻译成人能做决策的语言。

## 2.5 结构色路线里 cGAN 具体在做什么

### 2.5.1 它不是“直接生成图像”，而是“按目标颜色生成设计参数”

很多人一听 cGAN，会先想到图像生成。

但这个项目里的 cGAN 不是在生成图片，而是在生成设计参数。

更准确地说，是在生成：

- 底银厚度
- SiO2 厚度
- 顶银厚度

这三个值组成的设计向量。

所以你可以把它理解成一个“条件参数生成器”。

条件是什么？

是目标颜色的 Lab。

生成什么？

是可能实现这个颜色的厚度三元组。

### 2.5.2 为什么要用“条件”生成，而不是普通随机生成

因为系统不是想随便生成一些膜层组合，而是想生成“针对某个目标颜色”的组合。

如果没有条件输入，模型只能学整体分布：

- 什么厚度组合常见；
- 什么参数范围合理。

但有了目标 Lab 作为条件后，它才能学到：

- 如果目标偏蓝，厚度大致该往哪里去；
- 如果目标偏铜色，哪些参数区域更有机会；
- 同一目标下，哪些不同结构都可能成立。

所以“条件”是逆向设计的关键。

### 2.5.3 这个 cGAN 的网络结构大概是什么思路

从代码看，这个 cGAN 不是追求夸张复杂的大模型，而是偏轻量、可控、实验友好的结构。

主要部件有三个：

1. `Generator`
   生成器。输入目标 Lab 和随机噪声，输出归一化后的三维设计参数。
2. `Evaluator`
   判别器或评价器。判断生成的设计参数是不是像真实样本，并且当前是否匹配目标 Lab。
3. `LabRegressor`
   一个辅助回归器。它学的是“设计参数 -> Lab”，用来约束生成器不要只会骗判别器，还要真的接近目标颜色。

你可以通俗地把这个训练过程理解成三方博弈：

- 生成器负责提方案；
- 判别器负责挑毛病，说这方案像不像真实数据里的设计；
- 回归器负责看颜色，说这方案算出来像不像目标颜色。

这样一来，生成器受到的是双重约束：

- 不能乱编一个脱离真实分布的设计；
- 也不能只追求“像真实样本”，却离目标颜色很远。

### 2.5.4 为什么模型里还有噪声输入

因为这个问题不是单解问题。

如果生成器只有目标 Lab 这一个输入，它很容易学成：

- 同一个颜色永远输出一组最平均的答案。

但现实里，同一个颜色可能有多组不同的膜层方案。

所以加入噪声以后，模型可以在同一目标条件下生成不同候选。

这就更接近项目想要的效果：

**不是只给一个答案，而是给一组可比较的答案。**

### 2.5.5 为什么训练里又引入了 mode seeking

生成模型有个经典问题，叫“模式塌缩”。

通俗讲，就是：

- 明明一个目标颜色应该有很多可能结构；
- 结果模型老是输出几乎一样的那一两种结构；
- 表面看起来稳定，实际上失去了多样性。

项目里引入 `mode_seeking_weight`，就是在尝试缓解这个问题。

它本质上是在鼓励模型：

- 当噪声不同时，输出也要有适当变化；
- 不能把所有输入都压成一个模板答案。

这很符合结构色逆向设计的业务需求，因为这个项目明确要“多解候选”，不是只要“一个最像答案”。

### 2.5.6 为什么还要有 alpha ramp、显式学习率、判别器条件等训练旋钮

从训练脚本看，这个项目对 cGAN 训练不是一句“训练一下就完了”，而是做了大量可控实验设计。

比如：

- 生成器学习率
- 判别器学习率
- 每批判别/生成更新步数
- 判别器是否接收目标 Lab 条件
- 噪声维度
- mode seeking 权重
- 颜色误差附加权重
- alpha 从小到大缓慢拉升

这些旋钮背后的意思其实很朴素：

“这个问题并没有一个现成完美配方，所以项目在认真做训练稳定性调参。”

也就是说，项目不是把 cGAN 当黑箱，而是把它当成一个需要严密实验管理的研究对象。

## 2.6 论文复现路线到底在做什么

### 2.6.1 这不是随便训练一个模型，而是在对齐论文结果

项目里有一条非常明确的主线：先复现参考论文，再扩展到更大的业务目标。

这一步非常重要，因为它给项目建立了一个“客观参照物”。

如果不做论文复现，团队很容易陷入一种模糊状态：

- 训练能跑；
- 页面能展示；
- 结果看着好像也还行；

但没人知道到底离一个靠谱基线有多远。

而一旦对齐论文，就可以明确比较：

- 当前 mean best DeltaE 离论文有多远；
- 厚度恢复准确率离论文有多远；
- 多解分布是否像论文那样合理。

### 2.6.2 复现脚本不只是训练，还包括整套评估和产物导出

`backend/scripts/train_cgan_reproduction.py` 做的事情很多，不只是开个训练循环。

它大致承担了这些职责：

1. 读论文训练集和测试集
2. 构建设计参数和颜色样本
3. 启动 cGAN 训练
4. 训练 Lab 回归器
5. 记录损失
6. 周期性做 checkpoint 评估
7. 选出更好的 checkpoint
8. 在测试集上做 paper-style 评估
9. 计算多种指标
10. 导出图、表、日志、JSON 指标、候选 CSV、checkpoint 文件

换句话说，它不是“训练代码片段”，而是一个完整实验流水线。

### 2.6.3 当前复现关注哪些关键指标

项目里对论文复现的关注点并不只是单一的颜色误差，而是至少有以下几类指标：

1. `mean_best_delta_e`
   平均最优颜色误差。可以理解为：对每个目标，从生成的一堆候选里挑最好的那个，看整体平均效果。
2. `median_best_delta_e`
   中位数误差。避免极端值把平均数带偏。
3. `d2_ground_truth_within_5nm_ratio`
   特别关注中间层厚度恢复是否接近真实值。可以理解为“有多少比例在 5nm 误差以内”。
4. `solution_group_counts`
   解的分组数量。看模型是不是真的会给出多解，而不是只会重复一个模式。
5. `JSD`
   Jensen-Shannon Distance，用来比较生成厚度分布和真实测试分布的接近程度。

你会看到，这套评价体系其实是在同时回答三个问题：

1. 颜色像不像？
2. 参数准不准？
3. 解的分布像不像、多不多？

这就比只看一个损失值成熟很多。

### 2.6.4 为什么还要做 DBSCAN 聚类统计解的数量

因为这个问题不是说“多生成几个样本”就算多解。

假设你生成了 100 个候选，但它们都挤在厚度空间里非常靠近的位置，那本质上还是一类解。

所以项目里会用 DBSCAN 之类的聚类方法，把生成出来的解自动分簇。

这样它统计的就不是：

- “我一共生成了 100 个样本”

而是：

- “这些样本实际上形成了几类不同方案”

这个指标非常贴近论文里“一个颜色对应多个物理解”的核心主张。

### 2.6.5 为什么要支持不同检索指标对比

项目里不仅训练 cGAN，还会做检索基线对比，而且检索指标本身也支持两种：

- Lab 欧氏距离
- Delta E 2000

这背后有两个现实考虑：

1. 检索法是一个很强的简单基线。
   如果 cGAN 连检索法都打不过，那说明模型虽然复杂，但业务价值还不够强。
2. 检索指标不同，会直接影响种子解质量。
   如果 seed 选得更贴近人眼感知，后续 refinement 也可能更好。

所以项目里不是盲目“推神经网络”，而是一直拿简单基线做对照，这是一种很健康的工程/研究习惯。

## 2.7 为什么当前运行时算法是“检索 + cGAN + 物理复验 + 局部微调”的组合，而不是只用一种方法

这个组合看起来比“直接喂给网络出答案”复杂，但实际上很合理。

### 2.7.1 单独只用检索的问题

优点：

- 稳；
- 可解释；
- 不需要训练。

缺点：

- 只能从已有离散点里选；
- 分辨率受网格限制；
- 很难跳出采样密度的天花板。

### 2.7.2 单独只用 cGAN 的问题

优点：

- 有希望生成更灵活的候选；
- 更适合多解问题。

缺点：

- 训练不稳定；
- 有模式塌缩风险；
- 生成结果不一定物理上靠谱；
- 容易受训练数据质量、缩放方式、超参数影响。

### 2.7.3 单独只用局部优化的问题

优点：

- 结果可以很精细；
- 物理意义明确。

缺点：

- 需要好的初始点；
- 否则可能陷入局部最优；
- 计算成本会更高。

### 2.7.4 组合方法为什么更适合这个阶段

项目当前阶段采用的组合方法，本质上是在取三者优点：

- 检索提供稳定初始点；
- cGAN 提供更丰富的候选分布；
- 正向仿真保证物理一致性；
- 局部微调补足离散搜索和神经生成的精细度不足；
- 工艺扰动测试把“理论上可行”进一步筛成“工程上更可试”。

这非常像一个成熟设计系统的思路：

**不是押宝单一算法，而是让不同算法在不同环节干自己最擅长的事。**

## 2.8 结构色路线里“物理约束”具体体现在哪

很多人会以为，所谓物理约束就是“用了物理仿真”。其实没那么简单。

在这个项目里，物理约束至少体现在四层。

### 2.8.1 第一层：设计空间本身被限定

比如厚度范围不是无限的，而是明确限定在：

- Ag 层 10 到 30nm
- SiO2 层 60 到 180nm

这意味着系统不会乱跑到明显不在研究范围内的区域。

### 2.8.2 第二层：候选必须回到正向仿真里重算

也就是说，候选不是直接拿神经网络输出值交差，而必须重新过物理模型。

这让系统最终相信的是“仿真验证结果”，不是“神经网络自报成绩”。

### 2.8.3 第三层：颜色评价使用标准颜色学

如果只看原始参数距离或者随便用 RGB 差，这种“约束”并不可靠。

项目把颜色差评估放在 Lab 和 Delta E 上，其实是在提高物理结果到视觉结果之间的严肃性。

### 2.8.4 第四层：工艺扰动被纳入评分

这一步很重要。

很多仿真系统只关心理想解，但真实制造有误差。

项目通过 `+/-0.5nm` 扰动，把“制造误差”简化进了候选评分。

虽然这还不是完整工艺模型，但已经体现出一种工程意识：

**不是只求理论最优，而是求更稳妥的可制造解。**

## 2.9 当前结构色路线还没完全解决什么

要讲清项目，也必须讲它还没做完什么。

从现有文档和代码看，结构色路线已经能跑通，但还处在“原型可信、结果未最终收敛”的阶段。

主要还没完全解决的点包括：

### 2.9.1 cGAN 质量还没稳定压过基线

项目文档和算法总览里明确提到，当前 checked-in 的一些实验结果里，cGAN 还不能稳定打赢检索基线。

这意味着：

- 流程是通的；
- 训练是能跑的；
- 但模型还不是完全收敛的生产级答案。

### 2.9.2 当前结构还是比较低维的

现在的设计向量主要还是三层厚度。

这适合作为论文复现和 MVP，但离更复杂的真实微结构设计还有距离。

比如：

- 更复杂的周期结构；
- 更细的角度依赖建模；
- 更真实的材料色散；
- 更大空间范围内的连续结构约束。

这些还没有全部展开。

### 2.9.3 工艺约束还是简化版

当前对制造扰动的考虑主要是厚度微小偏移。

但真实工艺里还有更多问题：

- 横向不均匀；
- 边缘效应；
- 斜率限制；
- 连续性；
- 大尺寸拼接；
- 模具或曝光误差。

这些在文档层面有规划，但运行时还没有全部成为严格约束模型。

### 2.9.4 多视角、多图像、多像素级设计还没有完全走到终态

项目的总体目标里提到过更大尺度的：

- 多视角图像；
- 大尺寸灰度版图；
- 160k x 320k 输出；
- 0 到 7 微米高度范围。

但当前真正成熟跑通的部分，主要还是“单目标颜色 -> 紧凑参数向量”的路线。

这很正常，因为项目采取的是先把论文复现链条打稳，再往更大业务目标迁移的策略。

## 2.10 第二条算法路线：神经全息 / CGH / CITL 部分都做啥

前面讲的是结构色路线。下面讲另一条路线：神经全息。

这条路线的存在非常值得注意，因为它表明项目不是死盯一种物理机制，而是在把平台扩展到另一类光学设计问题。

### 2.10.1 这条路线的核心问题和结构色路线不一样

结构色路线的问题是：

- 给定想要的颜色；
- 反推出膜层参数。

神经全息路线的问题则更像：

- 给定目标图像或目标光场效果；
- 生成一个相位图；
- 通过 SLM 或传播系统重建出目标图像。

也就是说，这里设计对象从“膜层厚度”变成了“相位分布”。

### 2.10.2 当前代码里实现了哪些 CGH 基础算法

在 `backend/app/algorithms/cgh.py` 里，可以看到当前实现了几种典型的全息基线方法：

1. Gerchberg-Saxton
2. Wirtinger Holography
3. SGD CGH（带 Adam 风格更新）

这些方法的共同点是：

- 先给定目标振幅图；
- 再通过波前传播模型和迭代优化；
- 反推出一个相位图；
- 然后检查这个相位图在重建面上的效果。

这条路线和结构色路线最大的不同在于：

- 结构色主要处理“波长-厚度-颜色”的关系；
- CGH 主要处理“相位-传播-重建图像”的关系。

### 2.10.3 当前全息仿真在做哪种传播

代码里实现了两种传播模型：

1. Angular Spectrum Method，简称 ASM
2. Fresnel 传播

这两种方法都是波动光学里非常常见的传播近似方式。

用最通俗的话讲：

- 你可以把一张相位图看成一片特殊的波前；
- 光经过它以后，在传播过程中会变化；
- ASM 和 Fresnel 就是在计算这种变化；
- 最后你能在成像面上看到一个重建振幅图。

这套逻辑和结构色路线“算透射光谱再积分成颜色”完全不同，但底层精神是一致的：

**先用物理传播模型算出结果，再评价结果。**

### 2.10.4 为什么要从目标颜色生成一个简单目标图案

当前 `run_cgh_design` 里，为了做原型演示，它会把一个目标颜色先转成目标振幅强度，再生成一个简单图案，比如：

- Siemens 星
- bars
- rings

这不是生产终态，而是一个合理的原型设计方式。

原因很简单：

- 全息路线最终关心的是图像重建；
- 但在当前平台里，统一的业务输入还是一个 `targetHex` 颜色锚点；
- 所以为了把这条路线先跑通，系统用颜色亮度去驱动一个标准测试图样。

这相当于说：

“我先不要求用户上传一整张目标图，我先用统一接口把全息算法框架跑起来。”

这是一种很典型的工程分阶段策略。

### 2.10.5 Gerchberg-Saxton 在干嘛

Gerchberg-Saxton 可以非常通俗地理解为：

- 先随机猜一个相位；
- 把它传播到目标面；
- 强行把目标面的振幅改成你想要的振幅，但保留相位；
- 再反向传播回源面；
- 再强行把源面改成相位型约束；
- 重复很多次。

它是一个经典、直观、容易实现的相位恢复算法。

优点是：

- 基础；
- 稳；
- 常作为 CGH 基线。

缺点是：

- 质量不一定最好；
- 迭代结果有时有限；
- 对真实硬件失配没有主动建模。

### 2.10.6 Wirtinger Holography 在干嘛

它可以理解为一种更偏优化视角的方法。

大概逻辑是：

- 定义重建误差；
- 根据复数域梯度算出怎么改相位更能降低误差；
- 逐步迭代更新。

它比 GS 更像是在做连续优化，而不是简单投影交替。

在项目测试里，也专门比较了它与 GS 的效果，说明开发者希望知道：

- 更高级一点的优化方法，是否真的能带来更好的 PSNR/SSIM。

### 2.10.7 SGD CGH 在干嘛

这个方法更进一步，把相位优化写成类似深度学习里的随机梯度下降过程。

它：

- 定义损失；
- 算梯度；
- 做带动量和自适应项的更新；
- 最后得到相位图。

它的价值在于：

- 更容易和后续神经网络、CITL、可微代理模型接轨；
- 更像现代优化框架；
- 在原型里可作为“从经典迭代法过渡到神经优化法”的桥梁。

### 2.10.8 为什么全息路线里要看 PSNR 和 SSIM

因为这里的输出不再是单个颜色值，而是图像重建效果。

所以评价也从 Delta E 转成了：

- `PSNR`
- `SSIM`

通俗讲：

- PSNR 更偏“像素误差强弱”；
- SSIM 更偏“结构相似性”。

二者一起用，可以粗略判断：

- 重建图像亮度对不对；
- 结构形态保不保真。

### 2.10.9 为什么项目明确区分“仿真上限”和“真实设备表现”

神经全息路线里一个特别清醒的认识是：

**仿真图像漂亮，不代表上机也漂亮。**

这在全息领域非常关键。

因为真实系统里有很多理想模型里没有的东西：

- 光源不均匀；
- SLM 相位响应非线性；
- 像素串扰；
- 填充因子；
- 镜头像差；
- 对准误差；
- 相机采样误差。

所以项目文档和代码里反复强调：

- 纯仿真 PSNR/SSIM 只是上限；
- 真正生产质量必须靠 CITL；
- 最终相位图导出要被标定质量门槛约束。

这说明这条路线不是“加几个 CGH 函数凑热闹”，而是在认真向 Neural Holography 那套范式靠拢。

---

## 2.11 CITL 在这个项目里是什么意思

### 2.11.1 Camera-in-the-loop 的通俗理解

CITL，Camera-in-the-loop，最通俗的意思就是：

**不要只在电脑里比较“仿真结果和目标图”，还要把真实相机拍到的结果也拉回训练或优化回路。**

为什么这么做？

因为真实硬件和理想仿真之间有误差。

如果系统永远只跟仿真世界对齐，那它可能在电脑里很完美，上机却效果一般。

所以 CITL 的核心思想就是：

- 让相机拍到真实重建结果；
- 把这个真实结果和目标图比较；
- 再反过来修正模型或相位设计。

### 2.11.2 这条思路对项目意味着什么

它意味着这条路线最终不会停留在“纯算法演示”。

而是准备走向：

- 标定；
- bench capture；
- 物理闭环；
- 可微代理模型；
- 真实设备上的优化与验证。

项目当前代码里还没有把整个生产级 CITL 训练系统都写完，但已经把这条路线在：

- 数据结构；
- Agent 规划；
- 决策建议；
- 约束检查；
- 导出格式说明；

这些层面预留出来了。

这说明项目在架构上已经接受了一个事实：

**全息设计不是只靠离线仿真能闭环的，它天然需要硬件反馈。**

## 2.12 两条算法路线放在一起看，项目的算法侧到底长什么样

如果把结构色路线和全息路线放在一起看，这个项目算法侧的全貌可以概括成一句话：

**它在做一个“以物理正向模型为基础，以逆向生成和候选筛选为核心，以工程可验证性为目标”的光学设计平台。**

更展开一点说：

1. 它不是直接从文本到图纸的一步到位黑箱。
2. 它坚持先定义物理正向模型。
3. 它把 AI 放在逆向候选生成和效率提升的位置上。
4. 它把最终评估交回到物理或标准度量上。
5. 它接受一个目标对应多解，而不是硬凑单解。
6. 它开始重视制造偏差和真实硬件偏差，而不只看理想仿真。

所以如果有人问：

“这个项目算法部分到底在干嘛？”

最通俗的回答就是：

**它在学着像一个有物理常识、有实验意识的光学设计助手那样工作。**

不是只会算一个公式，也不是只会调一个神经网络，而是：

- 先建模；
- 再生成；
- 再验证；
- 再排序；
- 再交给人确认。

## 三、第二部分：软件工程化部分都做啥

前面讲的是“会算什么”。下面讲“怎么把它做成软件系统”。

这部分同样重要。

因为在真实项目里，算法能跑通，只是第一步。要让一个项目能长期推进，还需要一整套工程化支撑。

你可以把软件工程化理解成：

**把零散的算法能力，变成一个可运行、可管理、可维护、可展示、可验证、可复现、可扩展的系统。**

下面按层来讲。

---

## 3.1 为什么这个项目必须做工程化，而不是停留在 notebook 或脚本阶段

如果项目只是为了临时验证一个公式，脚本就够了。

但这个项目不是。

它要面对的问题包括：

- 算法有多种模式；
- 训练过程有很多实验配置；
- 运行时请求需要带业务上下文；
- 候选结果需要审阅；
- 产物需要管理；
- 文档和实验结论需要沉淀；
- 前后端需要共享统一数据结构；
- 后续还可能接 LLM 决策层、真实硬件、更多物理模式。

如果没有工程化，这些问题就会很快失控：

- 不知道哪个实验结果对应哪个模型；
- 不知道当前线上用的是哪份 checkpoint；
- 页面展示和后端输出对不上；
- 新人根本无法理解系统现状；
- 复现实验要靠口口相传。

所以工程化在这里不是“锦上添花”，而是项目从“可玩原型”走向“可持续迭代系统”的必要条件。

---

## 3.2 后端工程化：它把算法包装成了什么样

### 3.2.1 后端不是只暴露一个接口，而是组织成了清晰的职责分层

项目后端主要放在 `backend/app/` 下，整体采用的是 FastAPI 结构。

可以把它理解成几层：

1. 算法层
   - `algorithms/optics.py`
   - `algorithms/inverse_design.py`
   - `algorithms/cgan.py`
   - `algorithms/cgh.py`
   - `algorithms/colorimetry_data.py`
2. Agent/业务编排层
   - `agent.py`
3. 服务层
   - `service.py`
4. API 路由层
   - `api/routes/*.py`
5. 数据模型与 Schema 层
   - `schemas/*.py`
6. 运行时持久化层
   - `runtime_store.py`

这说明后端没有把所有逻辑都写进一个大文件，而是已经在做明确的职责拆分。

### 3.2.2 算法层负责“算”

这一层前面已经讲过，核心职责是：

- 光学正向仿真
- 逆向候选生成
- cGAN 训练与采样
- CGH 相位优化
- 颜色学换算

它尽量不关心页面长什么样，也不直接关心 HTTP 请求，而是专注在可复用的算法能力上。

这类拆分非常重要，因为算法应该能：

- 被 API 调；
- 被训练脚本调；
- 被测试调；
- 被其他业务编排逻辑调。

如果算法层和接口层粘死在一起，后面扩展会非常痛苦。

### 3.2.3 Agent 层负责“组织一次设计任务”

`backend/app/agent.py` 是一个很关键的文件。

它做的事情，不是新写一套物理算法，而是把各类能力串成一次业务运行。

例如它会负责：

- 读入用户设计请求
- 判断当前是 `structural-color` 还是 `neural-holography`
- 调用对应算法入口
- 构造运行摘要
- 选择当前活跃模型信息
- 生成时间线
- 生成决策建议
- 构造最终 `DesignRunResponse`

这层的角色很像“总调度员”。

你可以把它理解成：

- 算法层像工人；
- Agent 层像工头；
- API 层像前台窗口。

Agent 层的价值在于，它把一堆底层能力装配成“业务流程”。

### 3.2.4 API 路由层负责“把能力暴露出去”

当前后端 API 已经不是只有一个接口，而是分成多类路由：

- `/api/algorithm-overview`
- `/api/agent/design-run`
- `/api/runs`
- `/api/runs/{run_id}/workspace`
- `/api/runs/{run_id}/timeline`
- `/api/runs/{run_id}/candidates`
- `/api/runs/{run_id}/constraints`
- `/api/runs/{run_id}/artifacts`
- `/api/artifacts/{artifact_id}`
- `/api/project`
- `/api/health`

从工程角度看，这些接口的意义非常清楚：

1. `health`
   说明服务活着没有。
2. `design-run`
   用于真正提交一次设计任务。
3. `algorithm-overview`
   不是跑任务，而是看当前算法状态、实验状态、模型状态。
4. `runs/*`
   用于查看一次任务的运行结果。
5. `artifacts/*`
   用于查看与任务相关的产物。

换句话说，它已经不是“一个函数的网络封装”，而是在围绕“设计工作台”组织数据接口。

### 3.2.5 Schema 层负责“让前后端说同一种语言”

项目在 `schemas/` 里定义了大量 Pydantic 模型，比如：

- `DesignRequest`
- `DesignRunResponse`
- `CandidateSolution`
- `ConstraintCheck`
- `ExportEstimate`
- `WorkspaceDetail`
- `DecisionSupport`
- `AlgorithmOverview`

这些模型的价值非常大。

因为真实软件里最容易出问题的不是公式，而是接口契约：

- 前端以为有这个字段，后端没给；
- 后端把字符串改成数字，前端没跟上；
- 某个对象在不同接口长得不一样。

有了 Schema 层，就相当于先把“数据长什么样”说清楚。

这会带来几种好处：

1. 自动校验输入合法性
   比如 `targetHex` 必须匹配颜色格式。
2. 输出结构稳定
   前端可以放心消费。
3. OpenAPI 文档可导出
   便于自动生成类型。
4. 测试可以针对契约写得更清楚。

这类工作看似不显眼，但它是工程系统稳定性的基础。

## 3.3 运行时持久化：为什么项目专门做了 runtime store

### 3.3.1 一次设计任务不能只活在内存里

如果用户发起一次设计请求，系统返回完结果就完了，那么：

- 页面一刷新，结果可能就没了；
- 不能查看历史 run；
- 也没法做“产物详情页”。

所以项目设计了 `runtime_store.py`。

它做的事很实在：

- 把一次设计运行的工作区信息组织成 `RuntimeRunBundle`
- 保存到 `.runtime-store/runs/run-*.json`
- 支持重新加载
- 自动刷新 run 历史列表

通俗地说，它是在给原型系统加一层“轻量运行记录仓库”。

### 3.3.2 它保存的不只是最终答案，还保存工作区状态

`runtime_store` 存的不是一个简单“结果值”，而是一整套工作区数据：

- 当前 run 摘要
- 草稿信息
- 候选方案
- 约束检查
- 时间线
- 产物摘要
- 产物详情
- 模型信息
- 决策建议信息

这说明项目不是把任务当成“一次函数调用”，而是当成“一个可回看的工作对象”。

这对于后续做：

- 任务列表；
- run 回溯；
- 产物下载；
- 审批确认；

都是必要前提。

### 3.3.3 为什么这是一种很聪明的原型阶段工程做法

原型阶段如果一上来就接数据库，当然也可以，但成本更高。

现在这个做法的优点是：

- 比纯内存更稳；
- 比完整数据库更轻；
- 已经具备 run 持久化概念；
- 后续迁移到数据库也比较自然。

也就是说，它用很小的工程成本，把“任务状态持久化”这件事先建立起来了。

## 3.4 Algorithm Overview：这不是装饰页，而是“算法运维面板”

### 3.4.1 为什么项目专门做算法总览

很多算法项目只有一个“提交请求 -> 看结果”的页面。

但这个项目还多做了一个 `algorithm_overview.py`，并且开放 `/api/algorithm-overview` 接口。

这件事很重要。

因为项目本身存在很多“当前状态问题”：

- 目前最好的实验是哪一个？
- 当前活跃 checkpoint 是哪个？
- CUDA 是否可用？
- cGAN 当前质量离论文目标还有多远？
- 当前训练是不是 smoke run？
- 现在系统是否真的接了 LLM 决策代理？

这些信息不属于某一个业务 run，但又是使用系统的人非常关心的“全局状态”。

### 3.4.2 这个总览模块实际上做了什么

它会从 `backend/artifacts/` 下读取实验产物，自动整理出：

- 哪些实验目录存在
- 是否有 checkpoint
- 是否有 metrics
- 最佳实验是谁
- 最新实验是谁
- 当前实验属于什么阶段
- 主要指标是多少
- 当前环境是否能识别 GPU
- 当前代理配置是否启用

也就是说，它扮演的是：

**“算法系统状态解释器”**

而不是单纯数据转发器。

### 3.4.3 为什么这对团队协作特别有价值

如果没有这层总览，团队协作时会非常依赖口头同步：

- “你现在用的是哪个 checkpoint？”
- “最近最好的是哪次实验？”
- “现在系统默认走 fallback 还是 best model？”
- “GPU 这台机器上认出来了没？”

有了总览页之后，这些问题都能被系统自己回答。

这意味着项目开始具备一种很重要的工程属性：

**系统可以自描述自己的状态。**

## 3.5 前端工程化：它不是只做展示，而是在实现一个工作台

### 3.5.1 前端整体定位是什么

当前前端是 Vue 3 + TypeScript + Vite。

但更重要的不是技术栈名字，而是它的产品定位：

它不是普通的“结果页”，而是一个围绕设计任务构建的工作台。

从页面结构可以看出，前端想表达的是：

- 首页先看系统状态，再发起任务；
- 工作区页围绕一次 run 展开；
- 左边是 run 和产物；
- 中间是流程、时间线、文档；
- 右边是候选、约束、导出信息。

这个布局说明工程思路很明确：

**让用户像在一个设计控制台里工作，而不是像在一个表单网站里点按钮。**

### 3.5.2 首页在做什么

`HomePage.vue` 不是单纯欢迎页，它承担了两件关键任务：

1. 读取算法总览
   - 当前生产模型
   - AI 代理配置
   - 最佳复现实验
   - headline metrics
2. 发起设计任务
   - 填 requirement
   - 选目标颜色
   - 选设计模式
   - 选角度
   - 选偏振
   - 提交任务

这种设计很像真实工作里的第一步：

- 先看今天设备/模型/系统状态如何；
- 再决定要不要发起一单任务。

### 3.5.3 Workspace 页面在做什么

`WorkspacePage.vue` 是当前前端最核心的页面。

它在做的事情不是简单渲染一个对象，而是在把一次 run 组织成完整工作语境。

它会加载：

- 工作区详情
- 产物详情
- 当前选中产物
- 当前步骤状态
- 候选列表
- 约束列表
- AI 决策建议

你可以把它理解成：

**一次设计任务的全景视图。**

### 3.5.4 为什么页面里强调“步骤”

前端专门把工作流拆成：

- `brief`
- `generate`
- `select`
- `export`

这不是为了好看，而是在做显式工作流建模。

也就是说，前端不是把 run 当成一团静态数据，而是在说：

1. 先明确需求与条件；
2. 再生成和仿真；
3. 再挑方案；
4. 再导出。

这种阶段建模非常符合真实设计流程，也让页面更容易扩展成后续的审批、重跑、导出确认机制。

### 3.5.5 Inspector Panel 具体在干嘛

右侧 `InspectorPanel.vue` 是很典型的“工程化解释面板”。

它在不同步骤下会展示不同信息：

- 在简报/生成阶段，展示算法状态和全局指标；

- 在导出阶段，展示导出估算。

这说明前端不是单纯罗列信息，而是在根据当前任务阶段切换解释重点。

比如在选择候选时，它会特别展示：

- 目标颜色

- Delta E
- Composite score
- Manufacturability

### 3.5.6 Timeline 组件在干嘛

`AgentTimeline.vue` 的价值在于，它把系统内部过程变成了可叙述的事件流。

比如：

- 系统为什么选择了这条工作流

- AI 给了什么建议

这类信息不是纯算法指标，但对用户理解系统行为非常重要。

因为真实业务里，人常常会问：

- 你为什么给我这个结果？

- reference source
- target hex
- ranked under
- selected candidate
- active model
- export format

这本质上是在给产物建立“上下文身份证”。

未来如果要做：

- 导出归档；

- 审批留痕；
- 结果复盘；

这些 metadata 都会非常有用。

## 3.6 前后端契约工程：类型、接口、生成代码都在做什么

### 3.6.1 为什么这个项目重视 OpenAPI 和生成类型

项目里有：

- `backend/openapi.json`
- `backend/scripts/export_openapi.py`
- `frontend/src/api/generated-types.ts`

这说明它不是手写前端类型对着后端猜，而是在推动“接口文档自动导出、前端类型自动跟进”的工程链路。

这件事的价值非常大。

因为前后端协作最常见的问题不是大 bug，而是这种小错：

- 字段名写错；
- 可选字段理解不一致；
- 枚举值改了但另一边没同步；
- 某个响应结构被偷偷改了。

通过 OpenAPI 和生成类型，项目是在尽量减少这种“契约飘移”。

### 3.6.2 前端 service 层在做什么

`workspace-service.ts` 把所有接口调用统一收口：

- 列出 runs
- 获取 algorithm overview
- 获取 workspace
- 获取 artifact detail
- 创建 design run

这样做的好处是：

1. 页面不直接到处 `fetch`
2. 错误处理逻辑集中
3. API base URL 集中配置
4. 后续切换环境或扩展鉴权更容易

这就是典型的软件工程化动作：把“怎么调接口”从页面逻辑里抽出来。

### 3.6.3 错误处理为什么也算工程化

前端里有 `ApiClientError`，后端里有结构化 `ApiErrorPayload`。

这说明项目不是只追求“成功路径能跑”，而是在明确失败路径：

- run 不存在怎么办；
- artifact 不存在怎么办；
- 请求失败怎么报错；
- 前端如何解构错误 code 和 message。

成熟系统和 demo 的差别，很多时候就体现在这些地方。

## 3.7 LLM/Agent 工程化：这部分到底做了啥

### 3.7.1 项目里并不是“全靠大模型”，而是把大模型放在决策辅助层

从 `agent.py` 和配置逻辑看，当前系统对 LLM 的使用是比较克制的。

它不是让大模型直接生成膜层厚度，也不是直接替代物理仿真。

它更像在做这些事：

1. 生成一句简短的需求总结
2. 对候选结果给出决策辅助意见
3. 如果未配置 API key，则退化到启发式规则

这很合理。

因为在这种高物理约束问题里，大模型更适合做：

- 语言组织；
- 决策说明；
- 风险归纳；
- 人机协作桥梁。

而不适合直接替代数值仿真。

### 3.7.2 为什么要保留 heuristic fallback

这是很典型的工程思维。

项目没有假设“LLM 一定在线、一定可用、一定配置完整”，而是做了三种模式：

- `live`
- `fallback`
- `disabled`

如果没有 API key，系统不会崩，而是退到启发式决策支持。

这意味着：

- 页面还能工作；
- 工作流还能跑；
- 只是决策建议没那么智能。

这种设计非常务实，因为它避免了“整个平台被一个外部依赖卡死”。

### 3.7.3 决策支持输出为什么要结构化

项目要求决策层输出：

- `headline`
- `summary`
- `nextAction`
- `confidence`
- `recommendedCandidateId`
- `rationale`
- `risks`

这说明它不是把 LLM 当聊天机器人，而是把它当一个需要遵守合同的组件。

这种结构化输出非常重要，因为只有这样：

- 前端才能稳定显示；
- 产物 metadata 才能落地；
- 测试才能校验关键字段；
- 人机协作才能形成标准模板。

从工程角度看，这就是把“模型能力”变成“可集成能力”。

## 3.8 训练工程化：不是只有模型代码，还有实验管理和产物管理

### 3.8.1 为什么训练脚本这么长

`train_cgan_reproduction.py` 接近两千行，说明它承担的不是单纯训练。

它还在负责：

- 解析参数
- 选择数据源
- 选择实验预设
- 启动训练
- 定义 checkpoint 评估策略
- 实时写日志
- 输出 metrics
- 写 artifact manifest
- 写候选 CSV
- 写图
- 做 checkpoint 比较
- 兼容 GPU / CPU 环境

这说明项目已经进入一个典型阶段：

**真正难的不是“把模型写出来”，而是“把实验过程组织好”。**

### 3.8.2 为什么要有 experiment preset

脚本里预设了多种实验组合，比如 legacy tune4 alpha 等。

这很像实验管理里的“配方版本”。

它的作用是：

- 保留历史配置；
- 方便做消融；
- 避免每次都手输一长串参数；
- 提醒团队“本次实验在验证哪一个变量”。

这其实是在做轻量级实验规范化。

### 3.8.3 为什么 checkpoint 选择单独设计成评分策略

项目没有简单地“训练最后一轮就算最好”，而是引入 checkpoint 评估和选择。

这非常重要，因为：

- GAN 训练不一定越后面越好；
- 不同指标之间会冲突；
- 有时颜色误差更低，但分布更差；
- 有时厚度恢复更好，但多样性更差。

所以脚本里会综合：

- mean best Delta E
- median best Delta E
- d2 within 5nm
- JSD 等指标

做一个综合 checkpoint score。

这意味着系统在把“最好的模型”当成一个明确工程问题，而不是拍脑袋决定。

### 3.8.4 为什么 artifacts 目录是工程资产，不是杂物间

`backend/artifacts/` 下存了很多实验目录。

这些目录里常见内容包括：

- `metrics.json`
- `loss_history.csv`
- `candidate_samples.csv`
- `retrieval_metric_comparison.json`
- `generator_checkpoint.pt`
- `generator_checkpoint_best.pt`
- `analysis_summary.json`

这套结构的价值在于：

1. 实验结果可追溯
2. 算法总览可自动读取
3. 训练、评估、UI 可以共享同一份事实来源
4. 人可以后验分析，不必只盯控制台

也就是说，artifact 管理让项目逐渐具备“实验资产沉淀”能力。

## 3.9 测试工程化：项目都测了什么，说明了什么

很多原型项目只有代码没有测试，但这个项目显然不是。

从 `backend/tests/` 可以看到测试覆盖了多个方向。

### 3.9.1 光学仿真测试

例如：

- `test_optics_colorimetry.py`

这些测试在验证：

- 光谱积分是否符合参考计算

这说明项目在防止最底层物理/颜色学悄悄漂掉。

### 3.9.2 逆向设计运行时测试

例如：

- `test_inverse_design_runtime.py`

这些测试在验证：

- 是否返回了候选

- 角度变化是否真的影响结果

- retrieval metric 是否被正确报告

也就是说，测试不是只检查“函数跑没跑”，还检查“业务语义是否对上”。

### 3.9.3 cGAN 训练与复现辅助测试

例如：

- `test_cgan_reproduction.py`

它会测：

- loss 是否记录
- 输出是否在边界内
- 架构超参数是否被保存
- mode seeking loss 是否存在
- 若干复现辅助函数是否工作

这说明项目对训练基础设施也做了保障，不是完全靠人工盯实验。

### 3.9.4 CGH 仿真测试

例如：

- `test_cgh_simulation.py`

它验证：

- ASM / Fresnel 传播是否守恒或可逆
- 目标图像生成是否合理
- GS、WH、SGD 是否能产生有效相位图
- PSNR / SSIM 是否有基本门槛

这非常关键，因为它让第二条算法路线也不只是“有代码”，而是“有最小可信度校验”。

### 3.9.5 API 与工作区测试

例如：

- `test_workspace_api.py`
- `test_algorithm_overview_api.py`

这些测试在验证：

- 工作区接口是否返回完整字段
- 神经全息模式是否输出正确上下文
- 持久化 run 是否能重载
- 算法总览接口是否真的能从 artifacts 读出状态

这说明工程层和算法层不是分开各管各的，而是通过测试串起来了。

### 3.9.6 这些测试整体说明什么

说明项目在努力避免三类风险：

1. 数学正确性风险
2. 业务语义错位风险
3. 前后端契约漂移风险

这就是工程化最实在的价值之一：不是让系统“看起来更正规”，而是减少悄悄出错的概率。

## 3.10 文档工程化：这个项目不是“代码唯一真相”，而是有完整文档体系

### 3.10.1 文档不是附属物，而是系统的一部分

从 `docs/` 结构看，项目把文档分成了几个明确类别：

- `architecture/`
- `planning/`
- `decisions/`
- `design/`
- `references/`

这说明项目不是想到哪写到哪，而是在做知识分层。

### 3.10.2 各类文档分别承担什么角色

`architecture/`

- 讲系统怎么工作
- 讲模块边界
- 讲技术方案

`planning/`

- 讲要做什么
- 讲当前阶段
- 讲优先级和执行路线

`decisions/`

- 讲为什么这样选

`design/`

- 讲页面和交互
- 讲工作区体验

`references/`

- 讲论文、书籍、需求资料等参考源

这套分层很重要，因为它把“代码之外的知识”变成了可查找、可延续的工程资产。

### 3.10.3 ADR 为什么重要

像 `ADR-001`、`ADR-002` 这类文档，看似没代码，但其实非常关键。

因为一个项目里最容易丢失的不是“做了什么”，而是“为什么这么做”。

比如：

- 为什么用真实 D65 数据；
- 为什么 checkpoint 要加版本；
- 为什么当前优先走论文复现；
- 为什么前端是这种工作台结构。

这些如果只活在开发者脑子里，几个月后就会失传。

ADR 的作用，就是把这些关键理由钉住。

### 3.10.4 为什么说这对 AI 协作也有价值

因为这个项目明显是在一个“人和 AI 一起工作”的环境里推进的。

文档做得越清晰：

- 人越容易接手；
- AI 也越容易理解上下文；
- 后续改动越不容易偏题。

从这个角度看，文档工程化不只是对人有用，也是对智能协作系统有用。

## 3.11 配置与运行环境工程化：项目如何处理环境差异

### 3.11.1 为什么专门给 conda 环境文件

项目提供了 `environment.yml`，这意味着它希望环境尽量可重建。

原因很简单，这个项目依赖很多非纯前端库：

- NumPy
- SciPy
- PyTorch
- scikit-learn
- colour-science
- FastAPI

这类项目如果不固定环境，很容易出现：

- 我这里能跑，你那里不能跑；
- CUDA 版本不一致；
- 某个包升级后训练表现变了。

所以环境文件本质上也是工程资产。

### 3.11.2 为什么特别强调 GPU-aware

项目代码里对 PyTorch 设备选择做了明确处理：

- 自动检测 CUDA
- 支持手动指定设备
- 记录设备名称和数量
- 在算法总览里回显当前环境总结

这说明项目不是把 GPU 当“可有可无的加速器”，而是把“训练/推理当前落在哪种硬件上”当成运行时信息的一部分。

这很重要，因为对 cGAN 训练来说：

- CPU 和 GPU 不是一个量级体验；
- 同一个实验在不同设备上，成本和可行性差异很大。

### 3.11.3 为什么 CORS、API Base URL 这些小事也重要

后端配置了允许本地前端地址访问，前端则支持 `VITE_API_BASE_URL`。

这种事情看起来很小，但它体现出项目已经在考虑：

- 前后端分开启动；
- 不同环境地址切换；
- 本地联调体验。

这就是工程化的一个典型特征：

不是只有“大模块设计”，还包括大量让系统真正跑顺的“小连接件”。

## 六、结论：分别用最通俗的话总结两大部分

### 6.1 对“算法和物理机理仿真部分”最通俗的总结

这部分本质上在做三件事：

这部分本质上在做五件事：

1. 把算法包装成能通过 API 调用的服务。

4. 用前端页面把复杂算法结果翻译成人能理解的流程和卡片。
5. 用测试、文档、类型契约、运行时持久化保证系统能继续长下去。

如果再压缩成一句话，就是：

**它在把“会算的算法”做成“能长期协作和迭代的软件系统”。**

- 再用 cGAN 和逆向设计提升搜索能力；
- 再用工作台、时间线、候选排名、约束检查把结果变成可决策对象；
- 同时把未来的全息路线和硬件闭环路线也预埋进去。

这说明这个项目不是“东拼西凑的功能堆叠”，而是在形成自己的方法论：

**先用物理保证底座，再用 AI 提效，再用工程系统把结果变成可以持续使用的能力。**

如果后续继续推进，最值得优先投入的方向通常不是再做一层花哨界面，而是：

1. 继续提升论文复现质量，让 cGAN 在关键指标上更稳定超过简单检索基线；
2. 把当前紧凑参数逆向设计逐步迁移到更贴近真实制造的大尺度结构表示；
3. 在全息路线中补足真实 bench 标定和 CITL 闭环；
4. 让产物导出、实验记录、模型切换形成更完整的运维链条。

一旦这几步继续往前走，这个项目就会从“可信原型平台”逐渐变成“真正可用于光学设计协作的系统”。
